# 老王讲规划核心架构

## 1. 平台定位

**老王讲规划** 不只是一个填写页面并生成文档的工具，而是一个用于沉淀、表达、复用、分享并持续迭代工业网络规划经验的表达框架。

它想解决的问题是：

- 规划经验长期分散在个人记忆、案例文档、聊天记录与零散表格中，难以复用。
- 同一类项目中，不同人常常使用不同表达方式，难以比较，也难以沉淀。
- 输入、判断、推演、输出与经验回流之间，过去往往是割裂的。
- 项目做完之后，新的判断与新经验很难系统化反哺到下一轮规划中。

因此，该平台本质上是在建立一套统一的工业网络规划经验表达框架，使工业网络规划相关的资料采集、判断依据、设计推演、结果落地和经验迭代，能够在同一个平台中被统一承载。

当前首页已经围绕这套闭环，明确展示：
- 平台的闭环设计
- 平台的功能定位
- 五个一级菜单的职责说明

---

## 2. 核心五层

平台当前采用以下五层作为核心表达结构：

- **资料采集**
- **判断依据**
- **设计推演**
- **结果落地**
- **经验迭代**

这五层分别承担：

- `资料采集`：收集项目原始信息与约束条件
- `判断依据`：沉淀可复用的规则主题与判断逻辑
- `设计推演`：基于输入和规则形成具体设计结论
- `结果落地`：面向客户与内部整理正式输出
- `经验迭代`：将本项目中的新判断和新经验回流到经验库

---

## 3. 资料采集

这一层回答的是：

- 项目基础信息是什么
- 业务场景和对象构成是什么
- 安全需求和实施约束是什么
- 后续判断和推演的原始输入从哪里来

它承载的是所有规划工作的输入基础。

因此，这一层的本质是：

- **定义项目真实情况与原始约束。**

---

## 4. 判断依据

这一层回答的是：

- 在什么条件下该怎么选
- 为什么这么选
- 哪些情况下不适用
- 替代方案是什么
- 哪些经验可以跨项目复用

它承载的是可复用的规则主题与规则记录，而不是某一个具体项目本身的推导结果。

当前标杆经验体中，判断依据已按主题组织为：

- 拓扑规则
- 分层规则
- 区域规则
- 通道规则
- 边界互联规则
- 技术选择规则
- 地址规划规则
- 性能规则
- 稳定性规则
- 冗余规则
- 接入规则
- 选型规则
- 部署规则
- 安全规则
- 场景规则

因此，这一层的本质是：

- **定义项目判断所依据的可复用规则库。**

---

## 5. 设计推演

这一层回答的是：

- 基于已有输入和判断依据，如何一步步形成设计结论
- 设计原则如何收敛
- 架构划分如何推出来
- 技术选择、网络设计、工程设计如何逐步形成
- 哪些推演结论会形成新的经验回流线索

它承载的不是原始输入，也不是独立规则库，而是本项目中的设计推演过程。

当前系统已经在关键推演页中显式加入：
- `引用规则主题`
- `是否形成经验迭代项`
- `推演中的特殊说明`
- `建议沉淀内容`

因此，这一层的本质是：

- **定义项目如何基于输入与规则收敛为设计结果。**

---

## 6. 结果落地

这一层回答的是：

- 最终形成什么结果
- 给客户落成什么内容
- 给内部落成什么内容
- 哪些内容沉淀为案例和复盘材料

它承载的是正式输出层，包括：

### 面向客户
- 现场概述
- 设计说明
- 网络拓扑
- IP 规划

### 面向内部
- 规划复盘
- 案例提炼

因此，这一层的本质是：

- **定义项目结果如何被正式表达与交付。**

---

## 7. 经验迭代

这一层回答的是：

- 本项目里哪些判断超出了既有规则库
- 哪些内容值得沉淀为新的经验候选
- 这些新增内容应回写到系统的哪一层
- 哪些内容值得在后续项目中复用

它承载的是经验回流层，是平台从“规划流程系统”升级为“规划 + 经验回流闭环”的关键部分。

当前这一层主要包含：
- 特殊判断
- 经验候选
- 迭代总结

因此，这一层的本质是：

- **定义项目中新长出的经验如何回流并反哺经验库。**

---

## 8. 五层关系

这五层关系可以概括为：

1. 先通过 `资料采集` 收集项目真实情况与约束。
2. 再通过 `判断依据` 引入规划经验中的规则主题与规则记录。
3. 再通过 `设计推演` 形成项目中的设计过程与设计结论。
4. 再通过 `结果落地` 形成对内对外的正式输出。
5. 最后通过 `经验迭代` 把本项目中的新增判断与新经验回流到经验库。

这意味着平台已经不只是一个“出文档”的系统，而是一个：

- 输入收集系统
- 规则应用系统
- 设计推演系统
- 结果输出系统
- 经验回流系统

---

## 9. 当前系统映射

当前系统左侧固定骨架已经映射为：

- 资料采集
- 判断依据
- 设计推演
- 结果落地
- 经验迭代

这五层是平台的稳定骨架，不再依赖某一套具体经验体。

具体某套经验体的菜单、页面、字段、案例、输出和迭代内容，则由对应经验配置驱动。

---

## 10. 判断依据与设计推演映射

为了避免判断依据层过粗或过散，当前标杆经验体 `ISA95 + IEC62443` 已按“推演主决策点”建立规则主题映射。

### 10.1 一一对应主表

- 设计原则 → 场景规则 / 安全规则 / 技术选择规则
- 区域划分 → 区域规则 / 安全规则 / 场景规则
- 通道设计 → 通道规则 / 安全规则
- 边界互联 → 边界互联规则 / 安全规则
- 技术选择 → 技术选择规则 / 性能规则 / 稳定性规则
- 网络拓扑 → 拓扑规则 / 场景规则 / 冗余规则
- 分层设计 → 分层规则 / 场景规则
- 性能设计 → 性能规则 / 技术选择规则
- 稳定性设计 → 稳定性规则 / 冗余规则 / 部署规则
- 冗余设计 → 冗余规则 / 拓扑规则 / 稳定性规则
- 地址分段 → 地址规划规则 / 区域规则 / 边界互联规则
- 时延分析 → 性能规则 / 技术选择规则
- 性能评估 → 性能规则 / 稳定性规则
- 对象接入 → 接入规则 / 安全规则 / 区域规则
- 设备选型 → 选型规则 / 技术选择规则 / 性能规则
- 部署设计 → 部署规则 / 稳定性规则 / 实施约束相关规则
- 连接关系 → 通道规则 / 边界互联规则 / 冗余规则

### 10.2 一页可对应多类规则
- 推演页与规则页不是机械一一映射。
- 更合理的关系是：
  - 每个关键推演页有主规则主题
  - 同时允许引用 1–2 个辅助规则主题

### 10.3 当前系统中的显式链路
当前标杆经验体已经开始显式形成如下链路：

- 判断依据层定义规则主题
- 设计推演页显式记录“引用规则主题”
- 设计推演页显式记录“是否形成经验迭代项”
- 经验迭代层自动汇总推演页中的规则引用、回流标记与特殊说明

这意味着平台当前已经从“规则存在、推演存在、迭代存在”进一步升级为：

- **规则 → 推演 → 迭代** 的显式闭环
